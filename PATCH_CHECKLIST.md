# Обновление данных с новым патчем — чеклист

Единственный источник данных — `data.db` на сервере коллектора (`3.75.82.29`,
`~/counterplay-collector/data/data.db`). Коллектор непрерывно наполняет его по
всем эло и патчам. Всё ниже — производные от него выгрузки.

## Что обновляется

| # | Что | Скрипт (на сервере коллектора) | Куда | Автоматически? |
|---|-----|--------------------------------|------|----------------|
| 1 | **Данные драфта сайта** | `python3 export_draft.py --db data/data.db --out <site>/data/draft` | `stats.json` + `champions.json` → образ сайта | ручной деплой сайта |
| 2 | **Тир-лист сайта (все эло)** | `python3 export_tiers.py --db data/data.db --out <site>/data/draft` | `tiers.json` → образ сайта | ручной деплой сайта |
| 3 | **База программы** | `python3 publish_data.py --db data/data.db` (нужен `TMPDIR` на диске!) | GitHub-релиз `data`: тонкая `data.db` (177 МБ) + побакетные `data-<bucket>.db` (23–60 МБ) + `data-version.json` | **авто** после кругов сбора |
| 4 | **Руны/сборки** | `python3 export_runes.py --db data/data.db --out <out>` | том `/stats` сайта + `manifest.json` | ручной деплой в том |
| 5 | Иконки/имена | Data Dragon (авто в export_draft) | `champions.json`, программа | авто |
| 6 | Описания скиллов | отдельный скрипт | `abilities.json` | редко, вручную |

## Политика удержания патча (hold)

Новый патч НЕ становится основным (вес 1.0), пока по нему не набралось данных —
иначе тир-лист и рекомендации первые дни шумят. Реализовано в одном месте на
каждой стороне, пороги синхронны:

- `pipeline/freshness.py` — `pick_patches()` (используют export_draft / export_tiers);
- `RecommendationEngine.cs` — `PatchReady()` (движок программы);
- `bot/lib/freshness.js` — та же логика для проверки/показа.

Пороги: `READY_MIN_GAMES = 150` (взвеш. игр на «чемпион+роль»), `READY_FRACTION = 0.7`
(доля пар прошлого патча, покрытых новым), основной бакет — `emerald`.

## Проверка свежести

В Discord: **`/patch`** — показывает официальный патч (Data Dragon), готовность
нашей базы и что опубликовано на сайте / в программе / в рунах, с пометками
✅ актуально / ⚠️ отстало. Cron `patchWatch` (раз в час) сам пишет в
#announcements, когда вышел новый патч и когда по нему набрались данные.

```bash

#      скачать в локальный репозиторий сайта, затем:

ssh ubuntu@3.68.217.116 'cd ~/counterplay-site && docker compose build web && docker compose up -d web'

# 4 (руны): скопировать v1/<patch>/*.json + manifest.json в том:
#   ~/counterplay-site/stats/v1/  (manifest.json короткий TTL — обновится за ~10 мин)
```
Программа (пункт 3) обновляется сама: приложение при запуске сверяет
`data-version.json` и подкачивает свежую базу СВОЕГО эло (по рангу из LCU).

## Гигиена базы (авто)

`prune_db.py` — cron на хосте коллектора (ежедневно 04:00 UTC): держит последние
6 патчей во всех таблицах и 2 патча в `drafts`, чтобы серверная база не росла
бесконечно. Публикуемая копия дополнительно режет «длинный хвост» (пары с 1–2
играми) — их движок никогда не показывает. Разовый `VACUUM` для сжатия файла —
вручную при остановленном коллекторе.

**Что НЕ хранится:** в базе нет никаких персональных данных — ни ников, ни
puuid/summonerId, ни истории игроков. Только анонимные агрегаты по
чемпион/роль/эло/патч. `puuid` используются лишь на лету при сборе.
